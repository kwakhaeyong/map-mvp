## 현재 마일스톤: "결과 중심 구조 전환"

대화 중 실시간으로 자라는 지도를 없애고, 사용자가 "정리해줘"를 눌렀을 때 한 번에 완성도 높은 결과 화면(4블록: 요인+2x2매트릭스, 시나리오 비교, 타임라인, 통찰)을 만드는 구조로 전환하는 작업입니다. 6단계로 나눠 진행 중입니다.

| 단계 | 내용 | 상태 |
|---|---|---|
| PR1 | AI 되묻기(follow-up questions) 기능 | ✅ 머지·배포 완료 |
| PR2 | 최종 결과 생성 API 신설(Sonnet, 4블록 스키마) + 레이트리밋 | ✅ 머지·배포 완료 |
| PR3 | 결과 화면에 4블록 추가(기존 지도는 유지, 그 아래에 얹음) | ✅ 머지·배포 완료 |
| PR-A | (추가 개선) 2x2 매트릭스 라벨 겹침 해결 | ✅ 머지·배포 완료 |
| PR-B | (추가 개선) 블록별 재생성("다시 생성" 버튼) | ✅ 머지·배포 완료 |
| PR4 | 대화 중 실시간 지도 제거 + Bible §2/§6 갱신 | ⏸️ **대기 중** |
| PR5 | 나머지 안전장치 보강(아래 "미해결 리스크" 참고) | 부분 완료(레이트리밋·블록 재생성은 PR2/PR-B에서 앞당겨 반영) |

**PR4는 의도적으로 시작하지 않고 대기 중입니다.** 지금은 대화 중 지도(MapCanvas)가 여전히 안전망 역할을 하고 있고, 4블록 결과물의 실제 품질(문장 품질, closest_fit 판단의 타당성 등)은 이 개발 환경에 `ANTHROPIC_API_KEY`가 없어 검증하지 못했습니다. 오너가 프로덕션(`mapdecision.com`)에서 실제로 대화하고 "정리해줘"를 눌러본 뒤, 그 결과를 보고 PR4 진행 여부를 판단하기로 했습니다.

## CI / 브랜치 보호 · 머지 방식

- `.github/workflows/quality-gate.yml`("MAP Quality Gate")이 모든 PR에서 `typecheck`, `harness:check`, `design:check`, `build`를 실행하며, GitHub의 필수 상태 체크로 등록되어 있습니다.
- `.github/workflows/auto-merge.yml`("Guarded Auto Merge")이 Quality Gate 성공 시 **브랜치 이름과 무관하게 모든 PR을 자동으로 `main`에 스쿼시 머지**합니다. `chore/`와 `feature/`를 구분해 수동 머지하던 과거 정책은 더 이상 적용되지 않습니다 — 오너가 확인할 유일한 사람 검증 기회는 PR을 열기 전(또는 머지되기 전) 직접 화면을 보는 것뿐입니다.
- `main`은 PR을 통해서만 반영됩니다.

## 배포 · 모니터링

- `main`에 머지되면 Vercel이 자동으로 프로덕션에 배포합니다.
- `.github/workflows/production-smoke.yml`("Production Smoke Check")이 (1) `main` 푸시 시, (2) Guarded Auto Merge 성공 직후, (3) 6시간마다(cron) `mapdecision.com`이 정상 응답하는지 확인하고, 실패하면 GitHub 이슈를 자동 생성합니다.
- `.github/workflows/notify-n8n-on-merge.yml`이 머지될 때마다 n8n 웹훅으로 커밋 정보를 전달합니다. n8n 쪽에서 이 웹훅을 어떻게 처리하는지(알림 주기·채널 등)는 n8n 클라우드에 설정된 워크플로 내용이라 이 레포에서는 확인할 수 없습니다.
- 이 개발 샌드박스는 네트워크 프록시로 `mapdecision.com`에 직접 접근이 막혀 있어, 배포 확인은 위 GitHub Actions 결과(간접 증거)로만 합니다.

## AI / 안전장치 현황

- 대화 중 노드 추출과 되묻기(follow_up_questions)는 Haiku 4.5, 최종 결과 생성/블록 재생성은 Sonnet 5를 사용합니다. 두 경우 모두 `ANTHROPIC_API_KEY`가 없으면 대화는 규칙 기반(`local-conversation-provider`, `extractThinking`)으로 조용히 대체되지만, **최종 결과 생성은 규칙 기반 대체 결과가 없고 그냥 실패 처리됩니다** (아래 리스크 참고).
- 레이트리밋(모두 인메모리, 서버리스 재시작 시 초기화되는 느슨한 안전핀):
  - 하루 세션 시작 5회/IP (`registerSessionStart`)
  - 대화 메시지 40개/세션, 입력 5,000자/메시지
  - 최종 결과 생성(전체+블록 재생성 합산) 세션당 5회, 하루 10회/IP
  - 사용자 발화 3턴 미만이면 최종 생성 자체를 차단
- 법적 고지: `/privacy`, `/terms` 페이지와 AI 전송 고지 문구가 배포되어 있습니다(실무 참고용 초안이며 법률 자문 아님, 오너 확인 필요 문구 포함).

## 주요 미해결 리스크

1. **최종 결과 생성 실패 시 폴백이 거칠다.** `ANTHROPIC_API_KEY`가 없거나 Sonnet 호출이 실패하면 "지금은 결과를 생성할 수 없어요"라는 일반 메시지만 뜨고, 원래 설계에서 계획했던 "AI 연결 없음을 명시하고 규칙 기반 간이 결과라도 보여주는" 폴백은 아직 구현되지 않았습니다. 실패와 "AI 자체가 꺼져 있음"이 사용자 입장에서 구분되지 않습니다.
2. **4블록 결과의 실제 품질이 검증되지 않았습니다.** 이 개발 환경에 API 키가 없어 문장 품질, closest_fit 판단의 타당성, 톤(상담처럼 느껴지는지) 등을 실제로 확인한 적이 없습니다.
3. **결과 캐시와 대화의 불일치.** 결과 화면을 한 번 본 뒤 대화로 돌아가 더 말하고 다시 결과 화면에 오면, 자동으로는 예전 결과가 그대로 보입니다(블록별 "다시 생성" 버튼으로 수동 갱신은 가능).
4. **Bible §2/§6과 코드의 불일치.** 위 "제품 정의 기준 문서" 항목 참고 — PR4 전까지 의도적으로 남아있는 상태입니다.
5. **매트릭스 라벨 겹침은 8개 상한 안에서만 검증됨.** 요인이 8개를 넘는 경우는 스키마에서 막혀 있어 실제로 발생하지 않지만, 8개에 가까운 극단값에 대한 자동 테스트는 없습니다(수동 스크린샷 검증만 거침).
