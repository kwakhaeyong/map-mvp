# CLAUDE.md

## 프로젝트

**MAP OS (MapDecision)** — 사용자의 생각과 고민을 대화로 끌어내어, 글이 아니라 **시각화된 지도(MAP)**로 보여주는 의사결정 시각화 AI 플랫폼입니다.

> 정답을 대신 주는 AI가 아니라, 내 생각이 보이게 만드는 AI.

## 개발 원칙

1. **코드보다 UX/시각화가 먼저다.** 기능이 동작하는 것만으로는 완료가 아닙니다 — 실제 화면에서 "보기 좋고, 이해되고, 스크린샷 하고 싶은" 결과가 나와야 완료입니다.
2. **작게 개발하고, 자주 배포한다.** 큰 기능을 한 번에 만들지 말고, 눈에 보이는 작은 단위로 쪼개서 자주 내보냅니다.
3. **항상 Feature Branch + PR로 작업한다. main에 직접 커밋하지 않습니다.** 아무리 작은 수정이라도 브랜치를 새로 파고 PR을 엽니다.

## PR 작성 규칙

모든 PR 설명에는 다음을 반드시 포함합니다:

- **목적**: 이 PR이 왜 필요한지 (한두 문장)
- **변경 파일**: 무엇을 바꿨는지 파일 단위로
- **스크린샷**: 화면이 바뀌면 반드시 전/후 스크린샷 첨부 (모바일/데스크톱 둘 다 권장)
- **테스트 결과**: `typecheck`, `harness:check`, `design:check`, `build` 실행 결과 (통과/실패)
- **배포 상태**: Vercel Preview 링크와 확인 여부
- **리스크**: 이 변경으로 깨질 수 있는 부분, 알려진 한계

## "반영하자"의 의미

오너가 "반영하자"라고 말하면 **이 변경을 승인한다**는 뜻입니다. 하지만 이것이 **품질 게이트(typecheck·harness check·design check·build·시각 확인)를 생략해도 된다는 뜻은 아닙니다.** 승인은 "이 방향이 맞다"는 결정이고, 품질 검증은 별개로 항상 거칩니다.

## 기술 설명 방식

이 레포의 오너는 비개발자입니다. PR 설명, 커밋 메시지, 대화에서 기술 개념을 언급할 때는:

- 전문 용어를 그대로 쓰지 말고 쉬운 한국어로 풀어서 설명합니다. (예: "하이드레이션 에러" → "화면이 처음 그려질 때와 나중에 그려질 때 내용이 달라서 깜빡이거나 깨지는 문제")
- "왜 필요한지 / 사용자 입장에서 뭐가 바뀌는지"를 기술 구현보다 먼저 설명합니다.
- 코드 구조를 설명할 때도 비유나 화면 흐름 중심으로 풀어줍니다.
